1. PROBLEM STATEMENT
----------------------------------------------------------------

Designing and implementing a shared, real-time document editing
platform like Google Docs is complex due to concurrent access,
conflict resolution, version management, and scalability.

PROBLEM:
Multiple users need to simultaneously edit the same document
in real time. Edits from one user must be instantly propagated
to all other users. Conflicts arising from simultaneous edits
must be resolved automatically. Users need to be able to undo
edits, view version history, and restore previous states — all
without overwriting each other's work or corrupting the document.

SOLUTION SCOPE:
Build a full-stack real-time collaborative editor using
WebSockets for event-driven communication, with well-defined
design patterns managing state, behavior, and structure.

3. DESIGN PATTERNS USED
----------------------------------------------------------------

(a) OBSERVER PATTERN
    - Classes: DocumentSession (Subject), WebSocket clients (Observers)
    - How: All connected clients subscribe via subscribe(ws).
      When any client sends an edit, session.notify() broadcasts
      to all subscribers except the sender.
    - Why: Decouples the document state from the UI clients.
      Any number of clients can observe without the session
      knowing their implementation details.

(b) COMMAND PATTERN
    - Classes: EditCommand, CommandHistory
    - How: Every edit operation is encapsulated as an EditCommand
      object with execute() and undo() methods. CommandHistory
      maintains a pointer for traversal.
    - Why: Enables undo/redo without tight coupling. New command
      types (FormatCommand, InsertImageCommand) can be added
      without changing the history mechanism.

(c) MEMENTO PATTERN
    - Classes: Memento, MementoHistory
    - How: When a user clicks "Save Version", a Memento snapshot
      is created storing the full document content. Users can
      browse and restore any prior snapshot.
    - Why: Preserves encapsulation — the session's internal state
      is saved and restored without exposing implementation
      details to outside classes.

(d) STRATEGY PATTERN
    - Classes: ConflictResolver, LastWriteWinsStrategy,
      OperationalTransformStrategy
    - How: ConflictResolver holds a strategy reference. At
      runtime, the strategy can be swapped via setStrategy().
      Currently uses OT which detects whether server content
      diverged from the base before accepting client changes.
    - Why: New conflict algorithms (CRDT, timestamp-based) can
      be introduced without modifying ConflictResolver.

(e) SINGLETON PATTERN
    - Class: DocumentSessionManager
    - How: Private static _instance variable. getInstance()
      returns the same object every time. Constructor blocks
      duplicate creation.
    - Why: All WebSocket handlers must reference the same session
      store. Singleton guarantees one shared source of truth.

(f) FACADE PATTERN
    - Class: DocEditorFacade
    - How: Exposes two methods — handleClientMessage() and
      handleDisconnect(). Internally coordinates session
      lookup, conflict resolution, command execution, and
      observer notification.
    - Why: The WebSocket server only interacts with the Facade,
      not with individual subsystems. This hides complexity and
      makes the system easier to maintain and test.

4. DESIGN PRINCIPLES FOLLOWED
----------------------------------------------------------------

(a) Single Responsibility Principle (SRP)
    - DocumentSession: only manages observers and document state
    - EditCommand: only encapsulates one edit operation
    - MementoHistory: only stores and retrieves snapshots
    - ConflictResolver: only resolves conflicts
    Each class has exactly one reason to change.

(b) Open/Closed Principle (OCP)
    - ConflictResolver is open for extension (new strategies)
      but closed for modification.
    - New strategies like CRDTStrategy can be added by
      implementing the resolve() method — no existing code
      needs to change.

(c) Liskov Substitution Principle (LSP)
    - LastWriteWinsStrategy and OperationalTransformStrategy
      are interchangeable inside ConflictResolver without
      breaking any behavior.

(d) Interface Segregation Principle (ISP)
    - Observer subscription interface (subscribe/unsubscribe)
      is separate from the command interface (execute/undo).
    - Clients are not forced to implement interfaces they
      do not use.

(e) Dependency Inversion Principle (DIP)
    - DocEditorFacade depends on the abstract conflict
      resolution interface, not a concrete strategy.
    - High-level modules (Facade) do not depend on low-level
      modules (specific strategy implementations).

(f) DRY (Don't Repeat Yourself)
    - Notification logic is centralized in session.notify().
    - JSON parsing/error handling is in one place (Facade).

(g) Law of Demeter
    - DocEditorFacade does not reach into DocumentSession's
      internal command or memento objects directly — it calls
      session-level methods only.

5. ARCHITECTURAL PATTERN APPLIED
----------------------------------------------------------------

PRIMARY: EVENT-DRIVEN ARCHITECTURE
    - All communication between client and server happens via
      discrete events (JOIN, EDIT, SAVE_VERSION, CURSOR, etc.)
    - The server is reactive — it only acts when an event
      arrives. There is no polling.
    - Events are self-contained JSON messages with type,
      docId, userId, and payload.
    - This decouples producers (clients sending edits) from
      consumers (clients receiving broadcasts).

SECONDARY: MVC (Model-View-Controller)
    - Model: DocumentSession, EditCommand, MementoHistory
      hold the document state and business logic.
    - View: index.html + style.css render the UI.
    - Controller: app.js (EditorUI + CollabDocClient) handles
      user input and coordinates Model updates.

TERTIARY: CLIENT-SERVER with WebSocket
    - Persistent bidirectional WebSocket connection replaces
      HTTP polling.
    - Server maintains authoritative document state.
    - Clients are thin — they render what the server sends.

6. REFACTORING (DESIGN / CODE)
----------------------------------------------------------------

BEFORE REFACTORING (Monolithic approach):
    - One giant server.js with all logic inline inside the
      WebSocket onmessage handler.
    - Edit logic, conflict resolution, version saving, and
      notification all mixed in a single switch statement.
    - No separation of concerns; testing was impossible.
    - Duplicate code for finding sessions and broadcasting.

DESIGN REFACTORING APPLIED:

    Step 1 — Extract Observer
      Moved all client tracking into DocumentSession with
      subscribe(), unsubscribe(), notify() methods.
      Before: clients stored in a plain array, iterated inline.
      After: Set-based observer list with clean notification.

    Step 2 — Extract Command
      Before: Edits were applied directly as string assignments.
      After: Each edit wrapped in EditCommand; history tracked
      in CommandHistory with pointer-based undo/redo.

    Step 3 — Extract Memento
      Before: Versions stored as plain strings in an array.
      After: Structured Memento objects with userId, timestamp,
      and preview. MementoHistory enforces MAX_VERSIONS limit.

    Step 4 — Extract Strategy
      Before: Conflict resolution was an if-else block inline.
      After: ConflictResolver with swappable strategy objects.
      Adding CRDT now requires zero changes to existing code.

    Step 5 — Apply Singleton
      Before: Sessions stored in a module-level Map variable.
      After: DocumentSessionManager.getInstance() — guaranteed
      single instance, testable, injectable.

    Step 6 — Apply Facade
      Before: WebSocket handler was 150+ lines of mixed logic.
      After: DocEditorFacade.handleClientMessage() dispatches
      to the right subsystem. WebSocket code is 10 lines.

CODE REFACTORING APPLIED:
    - Replaced magic strings with named constants (message types)
    - Replaced nested callbacks with method extraction
    - Added input validation (JSON parse try/catch)
    - Removed duplicate session-lookup code
    - Added JSDoc comments for all public methods
    - Separated test file from production code
